That is an excellent review outcome. More importantly, the feedback he gave is exactly the kind of feedback a strong engineer would give: **your app works, but now you need to think about abuse, scalability, moderation, voice UX, and code structure.**

Here is how I would frame what happened and what you should do next.

## What the reviewer liked

Your **comments and README were useful**, not just decorative.

Your **LLM-as-judge component** was a strong design choice, because you separated the app into two LLM jobs: one model call for scoring and one for the next interview question.

## Reviewer feedback translated into engineering tasks

### 1. Add rate limiting / anti-spam protection

He is completely right. Right now, if someone clicks **Start interview** or **Submit answer** repeatedly, the app can trigger many API calls. That can burn through tokens and cost money.

The problem is:

```text
Button spam → repeated API calls → token/cost abuse
```

The simplest next protection is a **cooldown**:

```text
Allow one API action every X seconds
```

A better version is:

```text
Limit number of calls per session
Limit number of calls per minute
Disable button while request is running
Track last API call timestamp
```

You can explain it like this:

> The reviewer pointed out that my current app protects input quality, but not API usage. If a user repeatedly clicks the button, each click can create a new API call. My next improvement would be a rate-limit layer using session state, for example storing the last API call time and blocking requests that happen too quickly. In production, I would also add server-side rate limiting, not only frontend protection.

Important nuance: **Streamlit/session-state rate limiting is useful but not production-grade**. A malicious user could bypass frontend/session logic. Production-grade rate limiting usually needs server-side enforcement, user authentication, IP/session limits, or an API gateway.

### 2. Improve prompt-injection defense with a moderation model

Your current protection is keyword/pattern-based. It catches English phrases like:

```text
ignore previous instructions
reveal your system prompt
jailbreak
developer message
system message
```

The reviewer’s point is correct: a user could attempt the same attack in German, Lithuanian, Hindi, or with more subtle phrasing.

So the improved architecture would be:

```text
User input
↓
Moderation / jailbreak detection model
↓
If safe: send to main interview model
If unsafe: block or warn user
```

The Mistral Moderation 2 model he linked is specifically relevant because Mistral describes it as a moderation model with a **128k context window**, **jailbreaking detection**, and strong performance on **complex multilingual data**, including long multi-turn conversations. ([Mistral AI Documentation][1]) 

here is the Mistral Moderation 2 model: https://docs.mistral.ai/models/model-cards/mistral-moderation-26-03 

So your future answer can be:

> My current prompt-injection protection is rule-based, so it mainly catches obvious English attacks. A better version would first send user input to a moderation model like Mistral Moderation 2, which is designed for multilingual moderation and jailbreak detection. Only if the input passes that check would I send it to the main interview model.

This is exactly the right Sprint 2/next-version thinking.

### 3. Add voice input / speech-to-text

His voice suggestion also makes sense because this is an **interview practice app**. In a real interview, people speak. Typing answers is useful for testing, but speaking is more realistic.

The Voxtral link is relevant because Mistral presents Voxtral as an open-source speech-understanding model family, available in 24B and 3B variants, with multilingual capabilities and transcription/audio-understanding use cases. ([Mistral AI][2]) Voxtral also supports long-form context and multilingual speech capabilities, including German among the listed languages. ([Mistral AI][2])

Here is the Mistral Speech to Text model: https://mistral.ai/news/voxtral 

Future architecture:

```text
User speaks answer
↓
Speech-to-text transcription
↓
Transcript goes into the existing answer box / validation
↓
Same judge prompt scores the answer
```

Say it like this:

> Because this is an interview app, voice input would make the practice more realistic. I would add a speech-to-text step before the existing answer flow. The user records an answer, Voxtral or another transcription model converts it to text, and then the existing validation and scoring pipeline can stay mostly the same.

That is a very strong product answer.

### 4. Move prompts into a separate file

This is also good feedback. Right now, `PROMPTS`, `build_system_prompt()`, and `build_judge_prompt()` live in `app.py`. That is okay for a learning project, but as the app grows it becomes harder to maintain.

Better structure:

```text
app.py              → Streamlit UI and main flow
api.py              → OpenRouter calls
prompts.py          → prompt templates and prompt builders
validation.py       → input validation and prompt-injection helpers
scoring.py          → judge prompt, score extraction, feedback color
config.py           → constants and model names
```

For the prompts specifically:

```text
prompts.py
- PROMPTS
- build_system_prompt()
- build_judge_prompt()
- wrap_candidate_answer_for_model()
```

Review explanation:

> The reviewer suggested separating prompts from the main app logic. That makes sense because prompts are configuration-like assets. If I want to add a new prompt style or tune the evaluator, I should not need to edit the whole Streamlit app. Separating prompts into `prompts.py` would make the code easier to maintain, test, and refactor.
